# E3.4 — WFC Procgen + Per-Tile Sprites: Implementation Plan

**Status:** drafted, not started. Lands after E3.3 (hazard system).

## Goal

Replace the rectangular-rooms-and-corridors look with **per-tile authored data** so each cell of the dungeon can carry its own texture, walkability, LOS, and hazard properties. End state:

- **Walls, floors, and ceilings draw from per-tile texture lookups, not per-biome wholesale.** Two rooms in the same biome can have different walls — a frozen vault chamber with metallic alien plating next to a frozen corridor with ice walls.
- Multiple tile *variants* per category so a single room doesn't read as repeating wallpaper.
- Wave Function Collapse (or a simpler tag-based stamp system) decides which tile id goes in each cell, governed by per-biome adjacency rules and per-room style tags.
- Existing rectangular generator stays as a fallback flag for half-authored biomes.

## Why phased

The GDD self-flags WFC as "the highest-risk change … ~600 LOC." It's both an **algorithmic** investment (constraint propagation, backtracking, performance) AND an **authoring** investment (adjacency tables, tile sets per biome, three renderers picking up tile sprites). Trying to ship all of that in one PR is a multi-month tar pit.

This plan slices E3.4 into **four phases** so visible wins ship continuously and the WFC algorithm work happens last, against a system that already has tile data flowing through it.

| Phase | Ships | Effort |
|-------|-------|--------|
| 1 — Tile grid data model | Per-cell texture lookup + walls become tile data | 1-2 weeks |
| 2 — Per-room styling | Procgen tags rooms by style; walls/floors vary across rooms within a biome | 1 week |
| 3 — WFC algorithm | Adjacency-driven variant placement within rooms | 2-3 weeks |
| 4 — Editor support | Tile CRUD + adjacency table + style picker UI | 1 week |

The user-flagged "walls all have a single texture regardless of biome" complaint is fixed at the **end of Phase 1** (each cell has its own texture) and the **end of Phase 2** (different rooms within a biome look distinct). WFC adds coherence on top.

**What this plan deliberately does NOT do:** there's no intermediate "per-biome single wall texture" stage. That would bake in the wrong abstraction (biome ≠ texture key) and force a rewrite when per-tile lands. We go straight to the tile data model.

## Phase 1 — Tile grid data model + per-tile texture lookup

**Goal:** Walls become first-class tile data, not "absence of walkable." Every cell of the dungeon has an explicit tile id; the renderer looks up texture, walkability, and LOS per tile. This is the keystone phase — everything else builds on it.

### Data model

**Shared types** (`packages/shared/src/protocol.ts` + `content/types.ts`):

```ts
// Tile id 0 = void (out of bounds, no draw, blocks everything).
// Ids 1..N are biome-defined.
type TileId = number;

type TileGrid = {
  width: number;          // tiles
  height: number;         // tiles
  originX: number;        // world coords of cell (0,0) top-left
  originY: number;
  tileSize: number;       // px per cell (matches procgen TILE_SIZE)
  tiles: Uint8Array;      // length = width * height; row-major
};

// Per-biome tile registry, authored under
// packages/shared/content/biomes/<id>.json.
type TileDef = {
  id: number;             // unique within the biome
  label: string;          // 'frozen_wall_corner', 'floor_cracked', …
  role: 'floor' | 'wall' | 'door_frame' | 'pillar' | 'decoration';
  walkable: boolean;
  blocksLOS: boolean;
  blocksProjectiles: boolean;
  // Texture-override id. Resolves to /textures/biome_tile/<biome>::<id>.png
  // via getOverride('biome_tile', `${biomeId}::${tileId}`).
  textureId?: string;
  // Authoring tags for adjacency rules + room-style matching.
  tags?: string[];
  // Initial-placement weight when WFC picks among role-compatible tiles.
  weight?: number;
};

// Extends BiomeDef.
tileSet?: {
  tiles: TileDef[];
  // Symmetrical adjacency table by tag pair. Empty = anything goes
  // (Phase 1 default). Populated in Phase 3 to drive WFC.
  adjacency?: Array<{ a: string; b: string; allowed: boolean }>;
};
```

**SceneLayout extension** (`packages/shared/src/protocol.ts`):

```ts
// Optional during the migration. Pre-E3.4 saves omit it; renderers
// reconstruct a default grid from `walkables[]` (room interior =
// floor, everything outside = wall) when missing.
tileGrid?: TileGrid;
```

### Wire payload

70×70 dungeon × Uint8 = 4.9 KB per scene. Sent once on `welcome` / `scene_changed` alongside the other layout fields. Pre-existing layout is ~1-2 KB so the net inflate is small.

The biome tile registry already ships on welcome (it lives on BiomeDef, currently rebuilt for the wire in `getBiomesForWire`). Extend that wire shape with the tile set so client renderers can resolve a tile id → texture lookup without separate fetches.

### Procgen

**`packages/server/src/procgen.ts:generateFloorLayout`** gains a tile-grid emit pass after the existing rect placement:

1. Compute grid dimensions from `worldBounds` (round to TILE_SIZE).
2. Initialize all cells to `0` (void).
3. For each room: stamp its interior cells with the biome's default `floor` tile id, its perimeter cells with the biome's default `wall` tile id.
4. For each corridor: stamp interior with `corridor_floor` (biome can author it; falls back to floor).
5. For each door (placed by `generateLockedRoomMeta`): stamp the door tile under the building.

**Phase 1 scope is intentionally minimal:** every room of a given biome gets the *same* default floor + wall tile id. Visible result: walls now look different across biomes (because the tile id resolves to a per-biome texture), but uniform within a biome. Phase 2 introduces per-room variation.

### Renderer integration

All three renderers replace "is this position walkable?" + "is this a wall?" inferences with **tile-grid lookups at the cell containing the world-space position**.

#### FPS (`packages/client/lib/game/fps.ts`)

The raycaster's DDA already aligns to the tile grid (`TILE_SIZE = 32`). Refactor:

1. **Wall hit detection** — currently checks "is this cell inside any walkable rect?" Replace with: read tile id at the hit cell, check `tileDef.blocksProjectiles && !tileDef.walkable`. Equivalent for 1-tile-wall rooms; opens the door to interior pillars / partial blockers.
2. **Wall texture** (line ~817) — currently `getFpsOverrideTexture('building', texKind)` with texKind = 'wall' for procgen walls. New chain:
   - For player-placed buildings (`hit.isBuilding`): keep current `('building', kind)` lookup. Player walls aren't biome-driven.
   - For dungeon walls: read tile id at the hit cell → `('biome_tile', `${biomeId}::${tileId}`)` → fall back to `('biome_tile', `${biomeId}::default_wall`)` → fall back to the current `('building', 'wall')` if no biome tile set.
3. **Floor cast** — already uses `('biome_floor', biomeId)`. Extend to per-tile: at each row's sample point, read tile id → texture. Falls back to the existing single-floor lookup if the tile def has no `textureId`.
4. **Ceiling cast** — same as floor.

#### Iso (`packages/client/lib/game/iso.ts`)

Simpler than FPS but the same idea:

1. Wall sprite painting (`iso.ts:636`) — replace `getOverrideTexture('building', 'wall')` with the same precedence chain, keyed off the tile id at the wall's cell.
2. Floor — currently solid color from biome palette; extend to render a tile sprite per cell when a texture is authored. Cells without an authored texture keep the palette color (preserves Phase 1 → Phase 2 → … gradual authoring).

#### Top-down (`packages/client/lib/game/pixi.ts`)

**This is where top-down gains explicit walls for the first time.** Today, top-down fills walkables and lets walls appear as the *absence* of fill. New behavior:

1. Iterate the tile grid; for each non-void cell, draw a tile sprite (or palette-color rect when no texture is authored).
2. Walls render as actual pixels with their own texture rather than being negative space. Visible upgrade beyond just "they have a texture now."

Performance: 70×70 = 4.9K draws per frame on top-down. Use a tilemap renderer or persistent Container of sprites updated on `scene_changed` — *do not* rebuild every frame.

### Collision

`Scene.circlePassable` and friends currently use "in any walkable rect?" Switch to "tile at this cell is walkable?" Same answer for Phase 1 (rooms→walkable cells, walls→non-walkable cells), but the new check supports interior pillars / cross-room partial blockers without needing extra rect entries.

Keep the legacy `walkables[]` field on SceneLayout populated for backward-compat with anything reading it during the migration. Mark deprecated in a comment.

### Texture upload

Editor's master textures editor gets a "Biome tiles" section. Each row is `(biomeId, tileId)` pair, stored as `biome_tile/<biomeId>::<tileId>.<ext>`. The `id` segment uses `::` as separator since `<biomeId>::<tileId>` round-trips through the existing SAFE_ID regex (which already allows `_` and `-`; we add `:` to the allowed set OR encode the separator differently — encoding TBD when we touch the API route).

### Migration / fallback

- A biome with no `tileSet` keeps Phase 0 rendering: solid floor color, single global wall texture.
- A biome with a `tileSet` containing only `default_floor` + `default_wall` ships Phase 1's "biome-uniform" look. **The tile data path is exercised even when the visible result is "one floor, one wall per biome."** Phase 2 just adds tile id variety; the data model doesn't change.

### Files touched

- `packages/shared/src/protocol.ts` — `TileGrid`, `TileId`, `SceneLayout.tileGrid?`, biome wire-shape extension.
- `packages/shared/src/content/types.ts` — `TileDef`, `BiomeDef.tileSet`.
- `packages/shared/content/biomes/*.json` — initial `tileSet` with `default_floor` + `default_wall` per biome (single tile each).
- `packages/server/src/procgen.ts` — emit `TileGrid` alongside rect output.
- `packages/server/src/biomes.ts` — extend `getBiomesForWire` with tile set.
- `packages/server/src/scene.ts` — collision sampling reads from tile grid.
- `packages/client/app/api/editor/textures/route.ts` — allow `biome_tile` category.
- `packages/client/app/editor/textures/page.tsx` — "Biome tiles" section.
- `packages/client/app/editor/biomes/page.tsx` — tile-set authoring (lightweight Phase 1: just floor + wall texture upload per biome; Phase 4 brings the full CRUD).
- `packages/client/lib/game/fps.ts` — tile-id wall + floor + ceiling lookup.

### Acceptance criteria

- Author a `frozen_wall` and `frozen_floor` texture; descend into a frozen biome floor; FPS + iso + top-down all render the new textures.
- A biome without a `tileSet` keeps the Phase 0 look (no regressions).
- Movement, projectile collision, LOS all behave identically to Phase 0 on the same procgen layout.
- Wire payload size delta ≤ 6 KB per scene snapshot.

## Phase 2 — Per-room styling (different walls in different rooms)

**Goal:** Within a single biome, two rooms can look different. A frozen biome's vault chamber gets metallic alien wall tiles while a frozen corridor gets ice walls. **This is what the user actually asked for past the "biomes look distinct" baseline.**

### Mechanics

1. **Tags on rooms.** Procgen assigns a `roomStyle` per room at floor-generation time. Initial styles ship as 3-4 per biome:
   - Frozen: `cryo_vault` / `ice_corridor` / `glacier_chamber`.
   - Sun-Bleached: `plaza` / `tomb` / `marketplace`.
   - Catacombs: `crypt` / `passage` / `shrine`.
   - Alien Core: `nursery` / `conduit` / `sanctum`.
   - Default: `chamber` (single style).

   Style is picked deterministically from the floor's PRNG, weighted by per-biome style-frequency table.

2. **Tag matching.** Each tile in `BiomeDef.tileSet` carries `tags` (e.g., `['cryo_vault', 'metallic']`). Procgen, when stamping a room of style `cryo_vault`, picks tile ids whose tags include the style. This is deterministic — no algorithm yet, just filter + hash.

3. **Hazard category interaction (free for E3.3 cohesion):** extreme rooms automatically get the biome's "extreme" style if authored (`alien_core::sanctum`); safe rooms get the "safe" style. Authors don't have to wire this twice — the existing `roomCategories` from E3.3 informs default style selection.

### Data model addition

```ts
// On BiomeDef.tileSet:
roomStyles?: Array<{
  id: string;              // 'cryo_vault'
  label: string;
  weight: number;          // probability weight for non-special rooms
  // Optional bind to hazard category from E3.3. When the room's
  // category matches `forCategory`, this style is forced.
  forCategory?: 'safe' | 'corridor' | 'hazard' | 'extreme';
}>;

// On TileDef:
tags?: string[];           // includes style ids the tile is valid in
```

### Authoring story

The biome editor gains a "Room styles" sub-section in the tile editor. Each style has a list of tile ids that match (filtered by tag); authors don't manage adjacency yet, just "which tiles belong to which style."

### Procgen change

Tiny — replace the "use default_floor / default_wall" Phase 1 stamp with "filter by style tag, weighted-pick." Hash on `(worldSeed, cycle, floorIndex, roomIndex, cellX, cellY)` so each cell within a styled room picks a tile variant deterministically.

### Acceptance criteria

- A biome with 3 styles authored generates 3 visibly different room types per descent.
- Two players on the same world see the same room styles for the same `(cycle, floor)`.
- Extreme rooms (E3.3) auto-pick the extreme-bound style when one exists.

## Phase 3 — WFC algorithm

**Goal:** Adjacency-driven variant placement within rooms. Sand-blasted floor only neighbors sand-blasted walls; ornate floor never abuts industrial walls; debris piles cluster near walls instead of randomly.

### Algorithm sketch

Standard observation-based WFC:

1. **Initialize:** every cell starts as a superposition of all tile ids whose `role` matches the cell's category (floor cells → all floor tiles, wall cells → all wall tiles) AND whose tags match the room's style (Phase 2 filter).
2. **Pick lowest-entropy cell:** the cell with the fewest remaining valid tile ids. Tiebreak by deterministic PRNG.
3. **Collapse:** pick one of its valid tile ids weighted by `tile.weight`.
4. **Propagate:** update neighbors — remove tile ids whose tags don't satisfy the adjacency table against the collapsed tile's tags. Repeat until no more removals.
5. **Repeat 2-4** until every cell is collapsed.
6. **Backtrack on contradiction:** if a cell's superposition empties, undo the most recent collapse and try the next-best option. Cap backtracking depth; on cap, fall back to the Phase 2 weighted-pick output.

### Performance

Cell count ~5K for a 70×70 dungeon. Naive WFC: 1-3 seconds per floor. Acceptable for at-perihelion regen. Optimization later if needed: per-room WFC (split dungeon into rooms, WFC each independently with corridor stitching).

### Determinism

Same `(worldSeed, cycle, floorIndex)` → same tile placements. WFC's ordering and tiebreaks must use a single deterministic PRNG. No `Set` iteration order, no `Map` iteration order — all access by sorted keys.

### Fallback flag

`BiomeDef.tileSet.useWfc?: boolean` — defaults `false`. Biomes without WFC tuning keep Phase 2 weighted-pick output. WFC opt-in per biome until adjacency tables are authored.

### Files touched

- `packages/server/src/wfc.ts` — new file, ~400 LOC.
- `packages/server/src/procgen.ts` — call WFC after the rect+styles pass for biomes with `useWfc: true`.

### Risk registry

- "Why is this cell stuck?" debugging is the WFC tar pit. Mitigate: log cell index, neighbor ids, and which adjacency rule excluded what. Editor preview should optionally render entropy heatmaps so authors see contradiction zones live.
- Adjacency authoring is exhaustive in practice. Mitigate via tag-pair rules (Phase 1 design) so authoring is per-tag-pair, not per-tile-pair.
- Floor-regen latency. Mitigate: WFC runs server-side at perihelion, output cached in scene snapshot; clients never see the algorithm cost.

## Phase 4 — Editor support

**Goal:** Authors can add tiles, room styles, and adjacency rules without leaving the browser.

### Sub-route layout

`/editor/biomes/<id>/tiles`:

1. **Tile editor** — grid of tiles in the current biome. CRUD: add, delete, edit. Each row exposes id, label, role, walkable / blocksLOS / blocksProjectiles toggles, texture upload, tag chip input, weight slider.
2. **Room style editor** — list of styles. Each style picks a label, weight, optional `forCategory` binding (E3.3 hazard category), and a tag filter that selects which tiles match.
3. **Adjacency editor** — tag-pair table. Two columns of tag chips. Click a cell to toggle "allowed" for that pair. Default: allowed unless explicitly forbidden.
4. **Preview pane** — renders a generated dungeon with the biome's WFC output. Re-roll button bumps the seed; WFC re-runs client-side for instant iteration. Optional entropy-heatmap toggle.

### Files touched

- `packages/client/app/editor/biomes/[id]/tiles/page.tsx` — new sub-route.
- API route for biome content writes already handles JSON validation; add tiles / styles / adjacency to the schema.

## Walls — explicit treatment per phase

The user's flagged concern: "walls all have a single texture regardless of biome … should be per tile so different rooms/sections can have different walls."

- **Phase 1:** every wall cell has an explicit tile id resolved per-cell. The tile id determines the texture. **Walls can vary across cells in principle** — though Phase 1's procgen stamps the same default wall id throughout a biome, so visible result is "one wall texture per biome" until Phase 2.
- **Phase 2:** procgen tags rooms by style and picks wall tile ids that match the style tags. **Different rooms in the same biome look different.** Cryo vault walls vs ice corridor walls within the same Frozen biome.
- **Phase 3:** WFC picks the *specific* wall variant per cell so corner tiles, boundary tiles, decoration tiles cluster correctly. Adjacency rules ensure variants cohere across cell boundaries.

**There is no per-biome shortcut.** The plan goes straight to per-tile because per-biome bakes in the wrong abstraction.

## Decisions to make before starting

1. **Tile grid origin/size convention.** Top-left of cell `(0,0)` aligns to which world coord? Recommend: anchor to `worldBounds.x / worldBounds.y` rounded down to TILE_SIZE.
2. **Door tile vs door building.** Doors are currently buildings (procgen-placed, key-gated, indestructible). Either: (a) keep them as buildings and stamp a `door_frame` tile under them; or (b) migrate to a tile id and lose the BuildingState. **Recommend (a)** — minimal disruption.
3. **Tile id global vs per-biome.** Recommend per-biome — tile ids 1..N reused across biomes so each biome's set is self-contained. Wire payload encodes the biome alongside the grid.
4. **Adjacency authoring: tag-pair vs tile-pair.** Recommend tag-pair. Per-tile rules don't scale beyond ~6 tiles per biome; per-tag-pair scales to 50+ tiles with sane authoring.
5. **WFC scope: full-floor or per-room.** Recommend full-floor for the alpha. Per-room WFC is a perf optimization for later if regen latency hurts.
6. **Texture id separator.** `biome_tile/<biome>::<id>.png` collides with the existing SAFE_ID regex (`^[a-z0-9_-]+$`). Either widen the regex to allow `:` or encode the separator differently (`<biome>__<id>`). Recommend widening the regex; the existing path-traversal protection comes from category allow-listing, not character set.

## Out of scope for E3.4

These come later as separate roadmap items:

- **Per-tile sprite animations** (flickering crystals, shifting sand). Static sprites only.
- **Destructible terrain.** Walls remain indestructible structural; props handle the destructible roster.
- **Tile-level hazard zones.** E3.3 ships hazard at room granularity; per-tile hazard would let WFC place hot pockets within rooms — defer until balance is solid.
- **Multi-tile structures** (5×3 boss arenas pre-stamped over WFC output). Post-E3.4.
- **Vertical levels / pits.** All cells stay flat-floor in E3.4; vertical complexity is a separate axis.
- **Tile-level minimap.** Minimap stays room-level; tile-level would be too noisy for the player.

---

## Sequencing recommendation

**Phase 1 is the keystone.** Once tile-grid data is flowing through procgen, scene snapshot, wire, and all three renderers, every later phase is a localised algorithm or content change. Don't ship anything before Phase 1 — half-measures bake in wrong abstractions.

**Phase 2 is where the user-flagged complaint is fully resolved.** "Different rooms have different walls within the same biome" requires the per-room style pass. Phase 1 alone gets walls per-biome, not per-room.

**Phase 3 is optional polish.** If WFC turns out to be a multi-week tar pit, Phases 1+2 deliver ~80% of the visible payoff. WFC is coherence on top — important for a polished product, but not load-bearing.

**Phase 4 (editor) is parallel to Phase 3.** Authors can hand-edit JSON for Phases 1-2 if needed; the editor UI is a quality-of-life on top.

## What this unlocks

- **Visual identity per room within a biome** — different chambers feel different without needing entirely separate biomes.
- **Authoring biomes via the editor** rather than hand-edits across renderers.
- **Future scope easier:**
  - Per-tile hazard zones (E3.3 sequel) snap into the tile grid trivially.
  - WFC handles multi-tile structures (alcoves, columns) via "structure" tile sets.
  - Tile-level lighting (post-E3.4 polish) operates on tile grid coords.
  - Top-down view becomes properly textured for the first time.
