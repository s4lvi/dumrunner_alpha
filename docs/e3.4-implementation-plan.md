# E3.4 — Room Templates + Multi-Texture Biomes: Implementation Plan

**Status:** Phase 1 (tile-grid data model) shipped. Phases 2-4 below replace the previous WFC-based plan.

**What changed vs the original plan:** the original E3.4 staked the dungeon overhaul on Wave Function Collapse. Survey of shipping roguelikes (Spelunky, Dead Cells, Hades, Diablo, Enter the Gungeon, Returnal, Caves of Qud) found that essentially none use textbook WFC for level generation. The dominant pattern is **hand-authored room templates + procedural placement** — procgen picks rooms from a pool, the rooms themselves are designed by humans. WFC has a strong academic reputation and a thin shipping track record. Re-scoping E3.4 around what actually works in the genre.

## Goal

Each dungeon floor is assembled from **hand-authored room templates** drawn from a per-biome pool, connected by procedural corridors, painted with multi-texture biome surfaces. End state:

- A biome owns a pool of 8-20 room templates. Templates are JSON files with a tile grid + anchors for enemy / prop / loot / interactable spawns.
- Procgen picks templates by size + connectivity + role (regular / locked / boss / safe) and stamps them into the floor's tile grid.
- Within a template's interior, multiple wall and floor textures supply visual variety; per-cell weighted-random pick.
- Existing rect-room generator stays as a fallback for biomes with an empty pool, so each biome migrates independently.

## Why room templates, not WFC

| Aspect | WFC | Room templates |
|--------|-----|----------------|
| Geometry quality | Algorithmic, often muddy under combat scrutiny | Designer-authored, tactically intentional |
| Authoring | Adjacency tables (tag-pair, ~20-50 rules per biome) | Room JSONs (10-20 per biome) |
| Engine cost | ~600 LOC + propagation/backtracking + perf tuning | ~150 LOC stamp + connectivity matcher |
| Determinism | Hard — `Set` iteration order, backtracking edge cases | Trivial — pool index by seeded RNG |
| Failure mode | Cell stuck → backtrack tar pit | Pool exhausted → fall back to rect generator |
| Shipping track record in roguelikes | Thin (Caves of Qud edges; mostly indies) | Universal (Spelunky, Hades, Dead Cells, Diablo, …) |

Templates trade one kind of authoring effort (adjacency rules) for another (room layouts). The room layouts pay off twice — they double as the *combat scenes*, with sightlines and cover authored for tactical play, not just visual coherence.

## What Phase 1 already shipped (still load-bearing)

- `TileGrid` on `SceneLayout` (base64-encoded `tilesB64`).
- `TileDef` + `BiomeDef.tileSet` schemas; reserved tile ids (`VOID_TILE_ID=0`, `DEFAULT_FLOOR_TILE_ID=1`, `DEFAULT_WALL_TILE_ID=2`).
- `decodeTileGrid()` + `tileIdAt()` helpers in shared `geometry.ts`.
- `procgen.stampTileGrid()` emits per-cell tile ids from rect output.
- FPS + iso renderers read `('biome_wall', biomeId)` for non-building wall hits with fallback to `('building', 'wall')`.
- `biome_wall` texture category in the editor; biome editor exposes wall texture upload alongside floor / ceiling / skybox.

**Templates stamp into this same `TileGrid`.** Phase 1 wasn't wasted — it's the runtime data path templates emit through. The renderer's per-cell tile lookup is the contract templates produce against.

## Phase 2 — Multi-texture biomes (per-cell variant pick)

**Goal:** Each biome's walls and floors carry multiple texture variants. Per-cell hash picks one. No new geometry, no template engine — just visual richness on top of Phase 1's tile grid.

This is the cheaper of the two remaining wedges and ships independently of templates. **Lands first** so the visual win is in front of users while template authoring builds up.

### Data model

`TileDef.textureId` becomes `TileDef.textureIds: string[]` (array). Single-element arrays preserve current behavior.

```ts
// packages/shared/src/content/types.ts
export const TileDefSchema = z.object({
  id: z.number().int().min(1).max(255),
  label: z.string().min(1),
  role: z.enum(['floor', 'wall', 'door_frame', 'pillar', 'decoration']),
  walkable: z.boolean(),
  blocksLOS: z.boolean(),
  blocksProjectiles: z.boolean(),
  textureIds: z.array(idSchema).default([]),  // was textureId?: idSchema
  tags: z.array(z.string()).optional(),
  weight: z.number().nonnegative().optional(),
}).strict();
```

### Texture id convention

`biome_wall/<biomeId>__<variantIdx>.png` — underscore separator, no `:` (avoids the SAFE_ID regex change the original plan flagged at decision #6). `variantIdx` is an integer; the biome's wall TileDef's `textureIds` array indexes into them.

Falls back through:
1. `('biome_wall', `${biomeId}__${idx}`)` — the picked variant
2. `('biome_wall', biomeId)` — the Phase 1 single-texture key (back-compat for existing uploads)
3. `('building', 'wall')` — generic fallback

### Per-cell variant pick

Stable hash over `(worldSeed, cycle, floorIndex, cellX, cellY)` → index into `textureIds`. Implemented in shared so renderers and any preview tooling agree:

```ts
// packages/shared/src/geometry.ts
export function pickCellVariant(
  seed: number,
  cycle: number,
  floor: number,
  cellX: number,
  cellY: number,
  variantCount: number,
): number {
  if (variantCount <= 1) return 0;
  let h = seed >>> 0;
  h = (h ^ Math.imul(cycle | 0, 0x85ebca6b)) >>> 0;
  h = (h ^ Math.imul(floor | 0, 0xc2b2ae35)) >>> 0;
  h = (h ^ Math.imul(cellX | 0, 0x27d4eb2f)) >>> 0;
  h = (h ^ Math.imul(cellY | 0, 0x165667b1)) >>> 0;
  h ^= h >>> 16;
  return h % variantCount;
}
```

### Renderer change

FPS wall pass (`fps.ts:817`) — extend the existing biome lookup to include the variant index. Pass `worldSeed`, `cycle`, `floorIndex`, `cellX`, `cellY` into the wall pass; current code has tile cell `mx, my` already. `worldSeed`/`cycle`/`floorIndex` need to be threaded onto the layout (not currently exposed to the renderer — small wire addition or grab from welcome state).

Iso wall sprite (`iso.ts:636`) — same precedence; `tx, ty` are already in scope.

### Editor flow

Biome editor's "Wall" texture row becomes a small list field — author can add multiple variants. Each variant gets its own `TextureRow` keyed by `<biomeId>__<idx>`. Phase 1's single-row stays as the variant-0 default.

### Acceptance criteria

- Author uploads 3 wall textures for a biome; descend into a dungeon floor of that biome; FPS + iso show all three textures distributed across cells.
- Two players on the same world see identical variant placement for the same `(cycle, floor)`.
- Single-variant biomes (existing uploads) render unchanged.
- Wire payload delta: +0 bytes (variant index is computed, not transmitted).

### Files touched

- `packages/shared/src/content/types.ts` — `TileDef.textureIds`.
- `packages/shared/src/geometry.ts` — `pickCellVariant`.
- `packages/shared/content/biomes/*.json` — migrate `textureId` → `textureIds` arrays where authored.
- `packages/client/lib/game/fps.ts` — variant-index lookup.
- `packages/client/lib/game/iso.ts` — same.
- `packages/client/app/editor/biomes/page.tsx` — multi-row wall upload.
- `packages/server/src/world.ts` or wherever scene state ships — expose `worldSeed`/`cycle`/`floorIndex` to the wire so renderers can hash.

**Effort:** ~2-3 days.

## Phase 3 — Room template pipeline

**Goal:** Replace random rect placement with a template-driven floor generator. Each room is a hand-authored layout drawn from a per-biome pool.

### Data model

```ts
// packages/shared/src/content/types.ts
export const AnchorKindSchema = z.enum([
  'spawn',          // player arrival point (entrance template only)
  'extract',        // extract pad (entrance template only)
  'stairs_down',    // stairs to next floor (deepest template only)
  'enemy',          // procgen rolls a template's enemy roster here
  'prop',           // procgen rolls the biome's prop palette here
  'loot',           // scatter loot
  'door',           // locked-door slot
]);

export const RoomAnchorSchema = z.object({
  kind: AnchorKindSchema,
  // Tile coords relative to template origin (0,0 = top-left).
  tx: z.number().int().nonnegative(),
  ty: z.number().int().nonnegative(),
  // Optional override id (e.g. force a specific enemy template).
  // Empty = roll from biome roster.
  overrideId: idSchema.optional(),
}).strict();

export const RoomEdgeSchema = z.enum(['N', 'S', 'E', 'W']);

export const RoomTemplateSchema = z.object({
  id: idSchema,
  label: z.string().min(1),
  biomeAffinity: z.array(idSchema),     // which biomes can use this template
  // Tile grid dimensions.
  width: z.number().int().min(3).max(32),
  height: z.number().int().min(3).max(32),
  // Row-major byte array of tile ids (length = width * height).
  // Stored as base64 to keep JSON files compact.
  tilesB64: z.string(),
  // Which sides of the room have at least one corridor-connectable
  // floor cell. Procgen uses this to match templates against the
  // connectivity needs of the slot. Empty = template is a leaf
  // (boss/vault), can only attach via one explicit edge.
  entrySides: z.array(RoomEdgeSchema),
  anchors: z.array(RoomAnchorSchema),
  // Role tag for pool filtering. 'normal' = main pool; 'safe' /
  // 'extreme' / 'boss' / 'vault' restrict to specific slot types.
  role: z.enum(['normal', 'safe', 'extreme', 'boss', 'vault']),
  // Selection weight within the matching pool subset.
  weight: z.number().positive().default(1),
}).strict();
```

Stored as `packages/shared/content/rooms/<id>.json`. The id is global (a template can serve multiple biomes via `biomeAffinity`).

### Procgen rewrite

`generateFloorLayout` keeps its outer shape — same seed inputs, same `SceneLayout` output — but the room generation step changes:

1. **Plan the floor's room slots.** Reuse the existing rejection-sampling placement for `Rect[]` slots (gives us positions + sizes deterministically). Each slot has a target size range and connectivity requirement (which neighbours it must reach).
2. **For each slot, filter the template pool:**
   - `biomeAffinity` includes the floor's biome.
   - `role` matches the slot's hazard category (entrance → safe pool; stairs → normal/extreme; locked → vault).
   - `width <= slot.w && height <= slot.h` (templates can be smaller than the slot, never larger; surplus becomes corridor padding).
   - `entrySides` covers the slot's required corridor connections.
3. **Weighted-pick a template** from the filtered list using the seeded RNG.
4. **Stamp the template** into the floor's tile grid at the slot's origin. Anchors translate into the floor's anchor list.
5. **Corridors** generated as today between slot centers.
6. **Anchors** drive enemy/prop/loot placement instead of the current random-in-room scatter. `enemy` anchors roll the biome's enemy roster (preserves existing scaling). `prop` anchors roll the prop palette. `loot` anchors place scatter piles. `extract`/`stairs_down`/`door` resolve to interactables / building placements directly.

### Connectivity matching

The current procgen connects rooms in a strict chain (room `i` → room `i+1`). Templates with `entrySides: ['N', 'S']` work for a chain slot in the middle; templates with one entry side are leaves (boss / vault).

Phase 3 keeps the chain structure. **Multi-branch graphs are deferred**; not needed for templates to land. Side-branches are a separate procgen complexity.

If no template in the pool matches a slot's filter, **fall back to stamping a default rect template** (single floor + perimeter wall, no anchors). Logged at info level. This means a half-authored biome still produces a playable floor.

### Wire payload

Templates expand to tile data inline; **only the resulting tile grid + anchors ship to clients.** The template *library* never goes over the wire — clients don't need it. Same wire shape as Phase 1 plus an anchor list (~20-50 anchors per floor × 16 bytes ≈ trivial).

### Acceptance criteria

- Author 3 templates for a biome (entrance, normal, stairs); descend into a dungeon floor; the three rooms render with their authored layouts.
- Same `(worldSeed, cycle, floorIndex)` always picks the same templates and stamps them at the same positions.
- Removing every template from a biome's pool falls back to the existing rect generator without crashing.
- An anchor list emitted from the template drives at least one enemy spawn that previously came from random scatter.

### Files touched

- `packages/shared/src/content/types.ts` — `RoomTemplateSchema`, `RoomAnchorSchema`.
- `packages/shared/src/content/loader.ts` — load rooms from `content/rooms/*.json`.
- `packages/shared/src/protocol.ts` — `Anchor` type, `SceneLayout.anchors?` field.
- `packages/server/src/rooms.ts` — new file. `ROOMS` registry, `pickTemplate(slot, biome, rng)`, `stampTemplate(grid, template, originX, originY)`.
- `packages/server/src/procgen.ts` — call into rooms registry; emit anchors.
- `packages/server/src/scene.ts` — consume anchors for spawns (replaces some `generateInitialEnemies` / `generateInitialProps` random scatter).

**Effort:** ~5-7 days for the engine. **Authoring effort is separate and ongoing** — initial pool of 3-5 templates per biome is the bare minimum for a feature flag flip; 10+ per biome is the polish target.

## Phase 4 — Room editor

**Goal:** Authors create + edit templates from the browser. JSON files stay the storage format, but no hand-editing.

### Sub-route layout

`/editor/rooms` — list of all templates with biome filter, role filter, search. Click one to edit.

`/editor/rooms/<id>` — single-template editor:

1. **Tile palette** — list of valid tile ids drawn from the current biome's `tileSet`. Click a tile to select it as the paint brush.
2. **Tile grid canvas** — paintable HTML canvas, click/drag to paint cells with the selected tile. Right-click to erase to void.
3. **Anchor layer** — overlay on the grid. Click to drop an anchor (kind picker dropdown). Drag to reposition. Right-click to delete.
4. **Metadata** — id, label, biomeAffinity (multi-select), role, weight, entrySides (4 checkboxes).
5. **Preview** — small inline iso render of the template (reuse existing `IsoPreview`). Updates as you paint.
6. **Save** — POST to a new `/api/editor/rooms` endpoint, mirror of the existing biomes/enemies/props content APIs.

### Templates as biomes-pool members

The biome editor doesn't need to manage the template pool directly — `biomeAffinity` on each template already does this. The biome editor **lists** the templates that affinity-match it, with a count and links to edit, but the source of truth is the room JSON.

### Files touched

- `packages/client/app/editor/rooms/page.tsx` — list view.
- `packages/client/app/editor/rooms/[id]/page.tsx` — single-template editor.
- `packages/client/app/api/editor/rooms/route.ts` — list / save / delete.
- `packages/shared/src/content/loader.ts` — already touched in Phase 3.

**Effort:** ~5-7 days for a usable editor; another week for polish (undo/redo, copy-paste regions, template duplication).

## What's deliberately deferred

- **Sample-based WFC** (extracting adjacency rules from authored example patterns). Possible future polish *within* a template; not needed to ship the pipeline.
- **Template rotation / mirroring.** Doubling the pool via 4 rotations is appealing but the math interacts with anchor coordinates and entry sides. Author multiple variants by hand instead until pool size is the actual bottleneck.
- **Multi-floor templates** (templates that span two procgen slots). Useful for boss arenas. Phase 5 if needed.
- **Branching room graphs.** Today's procgen is a strict chain; templates work fine on it. Branching adds slot-graph complexity orthogonal to templates.
- **Per-tile hazard zones.** E3.3 hazards apply at room granularity. Template anchors could carry a `hazard_pocket` kind later; not blocking.
- **Vault / treasure-room special pools** beyond the basic role enum. The role enum supports `vault` from day one; richer rules (e.g. "vaults only appear past floor 5") are a tuning pass.
- **Authored encounters** — anchor definitions that prescribe specific enemy compositions instead of rolling the biome roster. Anchor schema already supports `overrideId`; using it deliberately is a content authoring choice, not an engine change.

## What this unlocks

- **Visual + tactical density without algorithmic risk.** Templates carry authored sightlines, cover positions, choke points. The engine work is straightforward stamping.
- **Independent biome migration.** A biome with no templates still uses the rect generator. Authors fill pools at their own pace; players never see "in-progress biomes" break.
- **Special-room pipeline for free.** The `role` field handles boss / vault / safe pools without separate code paths.
- **Author flow consolidates into one place.** Today, dungeons are a code change away. Post-Phase 4, every dungeon edit is a JSON edit through the browser.

## Decisions to make before starting

1. **Template grid size cap.** Recommend 32×32 tiles max per template. Bigger templates blow up the editor canvas and the procgen slot allocator.
2. **Anchor coordinate system.** Recommend: tile-coords relative to template origin (top-left = 0,0). World coords would couple anchors to placement, breaking template reuse.
3. **Tile id scope.** Recommend: tile ids stay per-biome, NOT per-template. A template references its biome's tileSet; templates can't ship custom tile ids. Means a frozen-biome template can't use sun-bleached tiles, which is fine — that's biome identity working.
4. **Pool emptiness behavior.** Recommend: silent fallback to rect generator with an info log. Throwing would block half-authored biomes.
5. **Determinism check.** All template selection uses the existing seeded `mulberry32` PRNG. No new randomness sources. Iteration order over the template pool is sorted-by-id (already established convention from `eligibleBiomeIds`).
6. **Save-game compat across phases.** Pre-Phase-3 snapshots have no anchors. Snapshot rehydration falls back to today's random-scatter spawn for those floors. New floors generated post-Phase-3 use anchors. Mark in the migration notes.

## Sequencing

1. **Phase 2 (multi-texture biomes) ships first.** Independent of templates; small visible win front-loaded while template authoring spins up.
2. **Phase 3 (template engine) without an editor.** JSON-authored templates by hand for the first few biomes. Engine validates, stamps, and runs.
3. **Phase 4 (template editor) parallel to Phase 3 polish.** Authors switch from JSON edits to the UI as soon as it's viable.

Each phase produces a shippable change. None depends on the next being done.

The data shape stays. The algorithm changes from "constraint propagation" to "stamp from a pool." The authoring shape changes from "adjacency tables + tile sets" to "room layouts + anchors." Net: simpler engine, more visible authoring, better-shipping pattern.
